从代码结构和项目架构来看，我梳理了三个方向的改进方案：

  ---
  1. 稳定性与健壮性

  LLM 调用链路当前问题：
  - generateStream() 使用 HttpURLConnection 原始 API，没有重试机制，网络抖动直接抛异常
  - generateResponse() 用 RestClient 也没有重试
  - 两个方法代码重复度高（构建相同 request body、解析相同响应）

  改进方案：抽取统一的 HTTP 调用层，接入 Spring Retry 指数退避

  llmService.call(messages)           // 统一入口，含重试
    ├─ llmService.callStream(messages) // 流式版本
    └─ llmService.callSync(messages)   // 同步版本

  结合 Ark API 429/502 的重试策略（CLAUDE.md 已提到但未实现）。

  当前隐患：Ark API 响应中 JSON 解析失败时走默认兜底值（"请继续阐述您的想法"），但如果连续失败，面试流会产出大量无意义问
  题，候选人体验极差。兜底应触发重试而非降级。

  ---
  2. 面试质量与智能性

  当前局限：
  - LLM 每次独立生成，没有全局的"能力地图"——不知道哪些知识点已覆盖、哪些还没问
  - 追问规则（熔断→阈值/幂等/退避）写在 prompt 里依赖 LLM 遵守，不可靠
  - 评分靠 LLM 主观打分，同一候选人不同 session 分数可能不一致

  改进方案：

  a) 面试能力矩阵：在 Redis 中维护一张覆盖矩阵，记录每轮问了什么能力点、候选人表现分（来自 LLM 评分）。LLM
  生成下一题时，把"弱项 top-3"注入 prompt，而不是只传 "askedQuestions" 列表。

  b) 评分校准：面试结束后，用第二次 LLM 调用综合全部对话重新评分（而非每轮累加），减少单轮评分偏差。

  c) 追问规则硬编码化：将 prompt 中的追问规则（熔断→阈值/幂等/退避）改为后端检测——用 NLP
  或关键词匹配识别候选人提到了关键术语（如"熔断"），然后自动在后端追加追问 prompt，而非依赖 LLM 自觉遵守。

  ---
  3. 工程架构

  代码组织：

  测试覆盖：项目目前零测试。核心逻辑（阶段推进、问题去重、兜底策略）应添加单元测试，LLM 响应解析应添加 JSON fixture
  测试。
